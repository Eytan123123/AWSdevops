מסמך ארכיטקטורה — מיגרציה ל AWS

DevOps Engineering Assignment · AWS Cloud Migration · Infrastructure as Code

1. תיאור רכיבי הארכיטקטורה

המערכת מורכבת מ-8 מיקרוסרביסים שמועברים מסביבת on-premise ל AWS. להלן תיאור כל הרכיבים וכיצד הם מתחברים.

VPC (Virtual Private Cloud)

ה VPC הוא הרשת הפרטית שלנו בתוך AWS — כמו ראוטר בבית שיוצר רשת פנימית. כל הרכיבים יושבים בתוכו ומבודדים מחשבונות אחרים.

הגדרנו טווח כתובות של 10.0.0.0/16 — כלומר כ-65,000 כתובות IP פרטיות לשימוש פנימי. הכתובות האלה לא קיימות באינטרנט הציבורי — אף אחד מבחוץ לא יכול להגיע אליהן.

ה VPC פרוס על 2 Availability Zones (מרכזי נתונים פיזיים נפרדים) — AZ1 ו AZ2 — כדי להבטיח שתקלה פיזית באחד מהם לא תוריד את המערכת כולה.

סאבנטים (Subnets)

בתוך ה VPC יש 6 סאבנטים — חלוקה פנימית של הרשת, כמו חדרים בתוך בית. כל סאבנט יש לו טווח כתובות וחוקי ניתוב משלו:

סאבנט CIDR AZ מה יושב בו
Public Subnet 110.0.1.0/24AZ1NAT Gateway, ALB node
Public Subnet 210.0.2.0/24AZ2NAT Gateway, ALB node
Private Subnet 1 (EC2)10.0.11.0/24AZ1EC2 Auto Scaling Group
Private Subnet 2 (EC2)10.0.12.0/24AZ2EC2 Auto Scaling Group
Private Subnet 3 (RDS)10.0.21.0/24AZ1RDS PostgreSQL Primary
Private Subnet 4 (RDS)10.0.22.0/24AZ2RDS PostgreSQL Standby

Public סאבנט ציבורי = בטבלת הניתוב שלו יש נתיב ל Internet Gateway — רכיבים בו יכולים לקבל ולשלוח תנועה לאינטרנט.

Private סאבנט פרטי = בטבלת הניתוב שלו יש נתיב ל NAT Gateway בלבד — רכיבים בו יכולים רק לצאת לאינטרנט (דרך NAT), אבל לא נגישים מבחוץ.

Route 53 (DNS)

שירות ה DNS של AWS. הוא יושב מחוץ ל VPC לחלוטין — תפקידו לתרגם שם דומיין לכתובת IP.

כשמשתמש מקליד כתובת כמו app.mycompany.com, שירות Route 53 מחזיר את כתובת ה IP של אחד מה ALB nodes (אחד לכל AZ). אחרי שהמשתמש הגיע ל node — ה ALB עצמו מחלק את התנועה בין ה EC2 instances הבריאים ב Target Group. כלומר Route 53 בוחר node, ה ALB בוחר EC2.

Route 53 לא מעביר תנועה — הוא רק עונה על השאלה "מה הכתובת?". התנועה עצמה עוברת דרך ה Internet Gateway.

User: "app.mycompany.com" → Route 53: "52.x.x.x" → User sends request to 52.x.x.x

Internet Gateway

השער בין ה VPC לאינטרנט. הוא מחובר ל VPC כולו ולא יושב בסאבנט ספציפי.

בלעדיו שום תנועה לא יכולה להיכנס או לצאת מה VPC. הוא לא מסנן ולא מנתב — רק "פותח את הדלת".

NAT Gateway

רכיב שיושב בסאבנט הציבורי ומאפשר לרכיבים בסאבנט הפרטי (EC2) לצאת לאינטרנט — מבלי להיות חשופים לתנועה נכנסת.

כשה EC2 רוצה למשוך Docker image מ ECR, הוא פונה ל NAT Gateway. ה NAT יוצא לאינטרנט עם ה Elastic IP (כתובת ציבורית קבועה) שלו. האינטרנט רואה את כתובת ה NAT בלבד — לא יודע שה EC2 קיים.

כשהתשובה חוזרת, ה NAT זוכר מי ביקש ומעביר אותה חזרה ל EC2 הנכון. אבל תנועה שמישהו מבחוץ יזם — נחסמת. זה מובנה ב NAT.

יש לנו 2 NAT Gateways — אחד בכל AZ — כדי שאם AZ אחד נופל, ה EC2 של ה AZ השני ימשיך לתפקד.

EC2 (10.0.11.x) → NAT Gateway → Internet Gateway → ECR / CloudWatch

Application Load Balancer (ALB)

מקבל את כל הבקשות מהמשתמשים ומחלק אותן בין ה EC2 instances.

הוא יושב בסאבנטים הציבוריים ויש לו כתובת DNS ציבורית אחת — אך מאחורי הקלעים יש לו node אחד בכל AZ (שתי נקודות כניסה לתוך המערכת). AWS מנתב את המשתמש אוטומטית ל node הבריא והפנוי.

אם AZ1 נופל — כל התנועה עוברת ל AZ2 בלי שהמשתמש ירגיש.

ה ALB גם מבצע health checks — כל 30 שניות הוא בודק כל EC2 instance. אם instance לא עונה — הוא מוצא מהרוטציה אוטומטית, והבקשות עוברות ל instances הבריאים.

Internet → Internet Gateway → ALB → EC2 instances

EC2 Auto Scaling Group

קבוצת שרתים וירטואליים (EC2 instances) שמריצים את ה Docker containers של המיקרוסרביסים.

ה Auto Scaling Group (ASG) הוא מנהל שיושב "מעל" ה instances ואחראי על להוסיף ולהסיר instances אוטומטית לפי עומס:

  • מינימום: 2 instances — תמיד אחד בכל AZ לזמינות גבוהה
  • מקסימום: 10 instances — תקרה למניעת עלויות

כל instance חדש שה ASG מרים נבנה לפי Launch Template — תבנית שמגדירה את סוג השרת, Security Group, IAM Role, ו user data (סקריפט שמתקין Docker ומריץ את הקונטיינר בהפעלה).

RDS PostgreSQL Multi-AZ

מחליף את מסד הנתונים ה on-premise. שירות מנוהל של AWS — כלומר AWS מטפל בעדכונים, גיבויים ותחזוקה.

ה Multi-AZ אומר שיש עותק סינכרוני (Standby) ב AZ השני שמתעדכן בזמן אמת:

  • Primary: מסד הנתונים הפעיל — ה EC2 מדבר רק איתו דרך כתובת DNS אחת
  • Standby: עותק מדויק — לא נגיש לאפליקציה, קיים רק כגיבוי

אם ה Primary נופל, AWS מקדם את ה Standby להיות ה Primary החדש אוטומטית תוך כ-60 שניות. כתובת ה DNS נשארת אותו דבר — ה EC2 לא צריך לשנות כלום וממשיך לעבוד בלי לדעת שמשהו השתנה.

שירותים תומכים

שירות תפקיד
Amazon ECR מאגר Docker images — ה EC2 שולף משם את תמונות הקונטיינרים בהפעלה
Amazon CloudWatch אוסף לוגים, מטריקות, ומגדיר התראות (CPU מעל 80%, שגיאות 5xx מעל 1%)
AWS Secrets Manager שומר את סיסמת ה RDS בצורה מאובטחת — ה EC2 שולף אותה בזמן ריצה, לאחר שה IAM Role שמחובר אליו מאשר את הגישה
Amazon S3 שמירת קובץ ה Terraform state — קובץ שמתאר את כל מצב התשתית
Amazon DynamoDB נעילת ה Terraform state — מונע מ-2 אנשים לשנות תשתית בו זמנית (נעילה אטומית)
AWS IAM ניהול הרשאות — Roles (תעודות זהות לרכיבים) ו Policies (מה כל רכיב מורשה לעשות)
Amazon Route 53 שירות DNS — מתרגם שם דומיין (כמו app.mycompany.com) לכתובת IP של ה ALB

2. החלטות עיצוב

למה EC2 ולא ECS או Lambda?

בחרנו EC2 כי הוא מתאים לפרויקט הזה — לא בהכרח כי הוא "הטוב ביותר" תמיד.

EC2 הוא שרת לינוקס רגיל — מתקינים עליו Docker ומריצים קונטיינרים ישירות. פשוט, שליטה מלאה, עקומת למידה קטנה.

  • לעומת Lambda:
    המיקרוסרביסים הם תהליכים שרצים כל הזמן ומחכים לבקשות. Lambda מיועד לפונקציות קצרות שמופעלות לפי אירועים, עם timeout של 15 דקות ו cold start — לא מתאים לשרת שצריך להיות זמין 24/7 עם 50,000 משתמשים.
  • לעומת ECS:
    ECS הוא שירות לניהול קונטיינרים — טכנית מתאים, ובפרויקטים עתידיים עם צוות מנוסה ב AWS הוא בהחלט אפשרות טובה. אבל הוא דורש ללמוד שפה חדשה: Task Definitions, Clusters, Services. EC2 פשוט יותר להתחיל איתו, נותן שליטה מלאה, ומתאים למיגרציה מהירה.

בנוסף, ההוראות מבקשות במפורש Launch Template עם user data להתקנת Docker ידנית — יכולת שקיימת רק ב EC2.

למה Multi-AZ RDS? מה זה מגן מפניו?

Multi-AZ מגן מפני נקודת כשל אחת במסד הנתונים:

  • תקלת חומרה פיזית ב AZ אחד — המסד עובר ל AZ השני
  • חלון תחזוקה — AWS יכול לעדכן את ה Primary, לעבור ל Standby, ולחזור
  • שגיאה אנושית שמורידה את ה instance

בלי Multi-AZ — כל תקלה במסד הנתונים מורידה את המערכת כולה ומפרה את ה 99.99% uptime SLA.

כיצד הארכיטקטורה משיגה 99.99% uptime?

99.99% uptime אומר פחות מ-52 דקות השבתה בשנה. שלוש שכבות הגנה מבטיחות זאת:

  • Multi-AZ EC2 ASG:
    תמיד רצים לפחות 2 instances — אחד בכל AZ. אם AZ שלם נופל, ה ALB מפסיק לנתב אליו תוך שניות, וה ASG מרים instances חדשים ב AZ הבריא.
  • Multi-AZ RDS:
    failover אוטומטי תוך ~60 שניות — AWS מקדם את ה Standby ל Primary בלי שום התערבות אנושית.
  • ALB Health Checks:
    בודק כל instance כל 30 שניות. instance שלא עונה מוצא מהרוטציה מיידית — משתמשים לא מגיעים ל instance שבור.

3. מודל רשת ואבטחה

מה יושב בפאבליק ומה בפרייבט?

הכלל הפשוט: אם הרכיב צריך לקבל תנועה ישירות מהאינטרנט — ציבורי. אחרת — פרטי.

רכיב סאבנט סיבה
ALB Public חייב לקבל תנועה ישירות מהאינטרנט — הוא "שומר הסף" היחיד
NAT Gateway Public צריך כתובת IP ציבורית (Elastic IP) כדי לצאת לאינטרנט בשם ה EC2
EC2 Private לא צריך להיות חשוף לאינטרנט — מקבל תנועה רק מה ALB
RDS Private מסד נתונים לא צריך להיות נגיש מבחוץ בשום מקרה

Security Groups — עקרון המינימום

כל רכיב מחובר ל Security Group שמגדיר בדיוק מה מותר ומה אסור ברמת הרשת (מי יכול לתקשר עם מי, על איזה פורט):

ALB Security Group

  • Inbound: פורט 80 (HTTP) מ 0.0.0.0/0 — כולם יכולים להגיע. בפרודקציה היינו מסיימים HTTPS עם תעודת ACM על פורט 443
  • Outbound: פורט 8080 לכיוון EC2 Security Group בלבד

EC2 Security Group

  • Inbound: פורט 8080 מ ALB Security Group בלבד — לא מהאינטרנט
  • Outbound: פורט 5432 לכיוון RDS Security Group
  • Outbound: פורט 443 לאינטרנט דרך NAT (עבור ECR, CloudWatch, Secrets Manager)

RDS Security Group

  • Inbound: פורט 5432 מ EC2 Security Group בלבד
  • Outbound: כלום — מסד הנתונים לא יוזם תנועה לשום מקום

הערה: ב Security Groups מציינים Security Group של הרכיב השני — לא כתובת IP. כך אם מחר יש 10 EC2 instances במקום 2, החוק תקף לכולם אוטומטית.

כיצד מאוחסנים ונגשים סודות?

סיסמת ה RDS לא כתובה בקוד ולא ב environment variables — היא נשמרת ב AWS Secrets Manager.

ה EC2 שולף אותה בזמן ריצה. כדי שזה יעבוד, ה EC2 מחובר ל IAM Role — תעודת זהות שמגדירה מה הוא מורשה לעשות. ה Role מכיל Policies שמתירות:

  • secretsmanager:GetSecretValue — לקרוא את הסוד הספציפי של סיסמת ה DB (לפי ARN — הכתובת הייחודית של המשאב)
  • ecr:GetAuthorizationToken, ecr:BatchGetImage — למשיכת Docker images מ ECR
  • logs:PutLogEvents — לשליחת לוגים ל CloudWatch

ב GitHub Actions — משתמשים ב OIDC לאימות מול AWS. במקום לשמור מפתחות גישה קבועים ב GitHub, GitHub מזדהה מול AWS ומקבל גישה זמנית בלבד — בטוח יותר.

4. אסטרטגיית סקיילינג

כיצד המערכת עוברת מ-1,000 ל-5,000 בקשות בשנייה?

ה Auto Scaling Group (ASG) אחראי על סקיילינג אוטומטי. הגדרנו Target Tracking Policy עם יעד CPU של 70%.

איך זה עובד:

  • ה ASG מודד את ממוצע ה CPU של כל ה instances ביחד בכל רגע נתון
  • אם הממוצע עולה מעל 70% — ה ASG מחשב כמה instances נדרשים ומרים אותם בבת אחת
  • אם הממוצע יורד מתחת — ה ASG מוריד instances לא נחוצים

הנוסחה:

Required instances = (Current avg CPU / Target CPU) x Current instances Example: Current avg CPU .... 90% Target CPU ......... 70% Current instances .. 2 90 / 70 x 2 = 2.57 → round up → 3 new instances

למה Target Tracking ולא Simple Scaling?

  • Simple Scaling:
    מוסיף instance אחד בכל פעם, מחכה כמה דקות לראות אם זה עזר, ואז מחליט אם להוסיף עוד — תהליך איטי.
  • Target Tracking:
    מחשב מיידית כמה instances נדרשים ומרים את כולם בבת אחת — תגובה מהירה לעומסי שיא.

הגדרות ה ASG בפרויקט:

  • min_size = 2 — תמיד לפחות instance אחד בכל AZ לזמינות גבוהה
  • max_size = 10 — תקרה למניעת עלויות בלתי מוגבלות
  • target CPU = 70% — שומר על 30% מרווח ביטחון לפני שה instance רווי

ה ALB והמשתמשים:

ה ALB מחובר ל Target Group — רשימה של EC2 instances שהוא מנתב אליהם תנועה. ה Target Group מחובר ל ASG, כך שכל instance חדש שעולה נרשם אוטומטית וה ALB מתחיל לשלוח אליו בקשות.

ה ALB עצמו הוא שירות מנוהל של AWS שמתרחב אוטומטית — 50,000 משתמשים בו זמנית מחולקים בין כל ה EC2 instances הבריאים בשני ה AZs, בלי שנצטרך לסקיל את ה ALB בעצמנו.
